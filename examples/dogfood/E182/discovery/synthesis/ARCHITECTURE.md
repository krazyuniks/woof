# ARCHITECTURE — E182

## Components

### 1. Checker registry (`woof/checks/registry.py`)

A Python module exporting a single `REGISTRY: dict[str, Check]` keyed by check ID. Each `Check` is a Pydantic model:

```python
class Check(BaseModel):
    id: str                    # e.g. "check_6_critique_blocker"
    stage: int                 # 1..6
    cost: Literal["cheap", "expensive"]  # cheap = no shell-out
    summary: str               # one-line human description
    runner: Callable[[CheckContext], CheckOutcome]
```

Stage-5 entries populate `id ∈ {check_1_quality_gates, check_2_outcome_markers, check_3_scope, check_4_contract_refs, check_5_plan_crossrefs, check_6_critique_blocker, check_7_commit_transaction, check_8_docs_drift, check_9_review_valve}`. Other stages have empty slots (registry has the structure; population deferred).

`runner` returns a structured outcome consumable by `check-result.schema.json`. Runners receive a `CheckContext` carrying `epic_id`, `story_id`, repo paths, and pre-loaded plan/critique objects — never an LLM-written prose blob.

### 2. `woof state classify`

```
woof state classify --epic <N> [--format json]
```

Filesystem-only classifier. Reads:
- `.woof/epics/E<N>/plan.json` (status of each story)
- `.woof/epics/E<N>/gate.md` (presence)
- `.woof/epics/E<N>/epic.jsonl` (tail; for last event type)
- `.woof/epics/E<N>/.last-sync` (presence + timestamp)

Returns:
```json
{
  "epic_id": 182,
  "stage": 5,
  "current_story": "S2",
  "has_open_gate": false,
  "next_action": "dispatch_story_executor"
}
```

No git, no network, no quality-gate runs. Used by `/wf` reconstitution and by `wf-run` between stories.

### 3. `woof check stage-N`

```
woof check stage-N --epic <N> [--story <Sk>] [--registry-id <check_id>] [--format json]
```

Iterates the registry filtered by stage (and optionally by `--registry-id` for single-check execution). Each runner produces a structured outcome; aggregator emits `check-result.json` matching `check-result.schema.json`. Exit code: 0 if all checks `ok`, 1 if any failed, 2 if an unrecognised registry ID was requested. `--self-test` enumerates the registry and exits non-zero on unimplemented entries.

The driver always invokes via `--format json`; humans use the prose default.

### 4. `woof gate write`

```
woof gate write --epic <N> --from-check-result result.json --position-file position.md
```

Mechanical gate construction:
- YAML front-matter built from `check-result.json` (gate_type derived from `triggered_by[0]`, `triggered_by[]` copied through, `opened_at` set to now, etc.).
- Prose body copied from `position.md`.
- Output written to `.woof/epics/E<N>/gate.md`.
- `gate_opened` event appended to `epic.jsonl` with the structured `triggered_by[]`.
- Validates output against `gate.schema.json`; non-zero exit on schema violation.

Skills (orchestrator at Stage 6, executor at exit) author `position.md` only. They never touch YAML.

### 5. Story executor (revised contract)

The `/wf:execute-story` skill body shrinks to:

1. Read story metadata, write code, run local quality gates iteratively until clean (executor's discretion — these are *not* the boundary checks).
2. Stage the diff with `git add <story.paths> .woof/epics/E<N>/{plan.json,critique/story-S<k>.md,epic.jsonl,dispatch.jsonl,audit/}`.
3. Write `executor_result.json` describing outcome.
4. Exit 0.

The skill body never enumerates the boundary checks, never writes `gate.md`, never invokes `git commit`.

### 6. Driver loop (revised, in `scripts/wf-run`)

Per story:
```
woof state classify --epic N → confirm next_action == dispatch_story_executor
woof dispatch claude --role story-executor --epic N --story Sk
# wait for subprocess
if exit ≠ 0:
    woof gate write --triggered-by subprocess_crash --exit-code <n>
    exit
read executor_result.json
case outcome:
    aborted_with_position:
        woof gate write --triggered-by executor_aborted --from-position position.md
        exit
    empty_diff:
        woof gate write --triggered-by empty_diff_review --from-position position.md
        exit
    staged_for_verification:
        woof check stage-5 --epic N --story Sk → check-result.json
        if check-result.ok:
            git commit -m "<subject>" -F <body from executor_result>
            log story_completed
        else:
            woof gate write --from-check-result check-result.json --position-file <auto-generated minimal position>
            exit
```

## Data contracts

### New schema: `check-result.schema.json`

```json
{
  "ok": true,
  "stage": 5,
  "epic_id": 182,
  "story_id": "S1",
  "triggered_by": [],
  "checks": [
    {
      "id": "check_1_quality_gates",
      "ok": true,
      "severity": null,
      "summary": "all 4 quality-gate commands exited 0",
      "evidence": null,
      "paths": [],
      "command": "just check-all",
      "exit_code": 0
    }
  ]
}
```

`severity ∈ {null, minor, major, blocker}`. `triggered_by[]` is the list of `id`s where `ok == false`. Top-level `ok = (triggered_by == [])`.

### New schema: `executor-result.schema.json`

```json
{
  "epic_id": 182,
  "story_id": "S1",
  "outcome": "staged_for_verification",
  "commit_body": "Implements check-result schema and the woof check stage-5 self-test runner...",
  "position": null
}
```

`outcome ∈ {staged_for_verification, aborted_with_position, empty_diff}`. `commit_body` required for `staged_for_verification`; `position` prose required for `aborted_with_position` / `empty_diff` (driver passes it to `woof gate write`).

### Gate schema (existing) extension

`gate.schema.json` `triggered_by[]` enum gains: `subprocess_crash`, `subprocess_aborted`, `executor_aborted`, plus the `check_<n>_<short_id>` IDs from the registry (auto-derived; the schema enum is generated from the registry at build time and snapshot-tested).

### Audit log additions

`epic.jsonl` events used by the new driver loop: `subprocess_dispatched`, `subprocess_returned`, `executor_result_recorded`, `check_stage_5_run`, `gate_opened`, `gate_resolved`, `story_committed`. Schemas live in `jsonl-events.schema.json` (existing).

## Stage-5 check inventory (canonical, replacing skill prose)

Verbatim from spark, registered as `check_1` … `check_9`:

1. **`check_1_quality_gates`** — run each `command` in `.woof/quality-gates.toml`; each must exit 0. *expensive.*
2. **`check_2_outcome_markers`** — for each `O<n>` in `story.satisfies[]`, regex-grep the staged test diff per `.woof/test-markers.toml`; ≥1 hit each. *cheap.*
3. **`check_3_scope`** — `git diff --staged --name-only` ⊆ `story.paths[]` (pathspec match), plus the allowed `.woof/` files. *cheap.*
4. **`check_4_contract_refs`** — `woof check-cd` (existing). *cheap-to-expensive depending on artefact tooling.*
5. **`check_5_plan_crossrefs`** — `plan.json` schema-valid; cross-refs intact (every outcome covered, every CD owned by exactly one story, deps acyclic, dep-target statuses honoured). *cheap.*
6. **`check_6_critique_blocker`** — `critique/story-S<k>.md` exists; schema-valid; top-level `severity` equals `max(findings[].severity)`; `severity != blocker`. *cheap.*
7. **`check_7_commit_transaction`** — staged set contains the four `.woof` durable files (`plan.json`, `critique/story-S<k>.md`, `epic.jsonl`, `dispatch.jsonl`) PLUS `audit/` artefacts per durable-state policy. No foreign `.woof/` paths staged. *cheap.*
8. **`check_8_docs_drift`** — per `.woof/docs-paths.toml` mappings (no-op when file absent). *cheap.*
9. **`check_9_review_valve`** — every-N stories AND end-of-epic; surfaces accumulated `severity: minor` findings via a `review_gate`. *cheap.*

## Bootstrap order (load-bearing)

The plan must order stories so that an executable, registry-driven Check 6 + driver-owned `git commit` land in the **first** story. Subsequent stories execute under the new safety. Sequence sketch:

1. **S1 (mandatory first):** Land `woof check stage-5 --story --registry-id check_6_critique_blocker` + the registry skeleton + driver invocation in `scripts/wf-run` that runs `git commit` only when check-result.ok is true. This single story replaces the broken Check 6 prose path.
2. Subsequent stories populate the remaining 8 check runners, the schemas, the snapshot tests, and the executor contract revision in any topologically valid order.

The orchestrator (this skill) and the planner skill must enforce this constraint at plan critique time.
